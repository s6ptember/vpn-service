# Деплой на VPS

Пошаговый разбор: от пустого сервера до работающего мини-аппа, продающего подписки.

Архитектура и её обоснование — в `tech.md` (раздел 3). Здесь только порядок действий.

Ориентир по времени: **40–60 минут**, из них половина — ожидание DNS и сборки.

---

## Что понадобится заранее

|                  | Что именно                                     | Где взять                                |
| ---------------- | ---------------------------------------------- | ---------------------------------------- |
| VPS              | 2 vCPU, **4 ГБ RAM**, 40 ГБ NVMe, Ubuntu 24.04 | любой провайдер                          |
| Домен            | `vpn-service.help` с доступом к DNS            | регистратор                              |
| Бот              | токен и username                               | [@BotFather](https://t.me/BotFather)     |
| Свой Telegram id | число, не username                             | [@userinfobot](https://t.me/userinfobot) |
| Stripe           | секретный ключ и вебхук                        | дашборд Stripe                           |

**Про 4 ГБ.** В работе стек занимает около 500 МБ — но образ приложения собирается **на самом
сервере**, и `vite build` в пике съедает 1.28 ГБ (измерено на этом проекте). На 1 ГБ сборка упадёт
с OOM. Минимум, на котором получится, — 2 ГБ плюс 2 ГБ swap.

Почему нельзя собрать образ в CI и просто скачать: `config.ts` читает `$env/static/private`, и vite
вмораживает значения секретов в `build/` на этапе сборки. Образ **сам является секретом** — его не
пушат в реестр. Если 4 ГБ взять негде, соберите образ у себя и перенесите напрямую:

```bash
docker save vpn-service-app:local | ssh user@vps 'docker load'
```

**Stripe.** Если аккаунта ещё нет (для ряда стран это блокер на недели — см. `tech.md` 17), деплой
не отменяется: поставьте `PAYMENT_PROVIDER=fake`, и заработает всё, кроме приёма реальных денег.

## Шаг 1. Подготовить сервер

```bash
ssh root@<IP-сервера>
```

**Docker:**

```bash
curl -fsSL https://get.docker.com | sh
docker compose version   # должно быть v2.x
```

**Swap** — 2 ГБ, страховка на время сборки:

```bash
fallocate -l 2G /swapfile && chmod 600 /swapfile && mkswap /swapfile && swapon /swapfile
echo '/swapfile none swap sw 0 0' >> /etc/fstab
```

**Фаервол:**

```bash
ufw allow 22/tcp
ufw allow 80/tcp
ufw allow 443
ufw allow 8443/tcp
ufw enable
```

`443` без указания протокола — это и tcp, и udp; udp нужен для HTTP/3. `8443` — трафик VLESS.

**Порт `8000` не открывайте никогда.** Это панель Marzban: она привязана к петле хоста и
достижима только через SSH-туннель. Открыть его наружу — значит выложить дашборд, `/api` и выдачу
токенов в открытом HTTP мимо Caddy.

## Шаг 2. Настроить DNS

Две записи типа `A` на IP сервера:

| Type | Name  | Data           | TTL |
| ---- | ----- | -------------- | --- |
| A    | `app` | `<IP-сервера>` | 600 |
| A    | `sub` | `<IP-сервера>` | 600 |

**TTL 600, а не час.** Пока настраиваете, ошибка в записи будет стоить 10 минут вместо часа.

`AAAA` не добавляйте, если у сервера нет IPv6: Caddy попробует выпустить сертификат и по IPv6, и
запись «в никуда» сломает выпуск.

Дождитесь распространения и **проверьте до того, как что-то запускать**:

```bash
dig +short app.vpn-service.help
dig +short sub.vpn-service.help
```

Обе команды должны вернуть IP сервера. Если возвращают пусто или чужой адрес — ждите дальше. Caddy
идёт за сертификатом сразу при старте, а Let's Encrypt лимитирует неудачные попытки.

## Шаг 4. Заполнить `.env`

`.env` не хранится в git, поэтому после `git clone` его на сервере нет.

**Вариант А — сгенерировать на месте:**

```bash
scripts/init-env.sh
```

Скрипт создаёт `.env` из примера и сам генерирует то, что можно сгенерировать: `SESSION_SECRET` и
оба пароля Marzban. Остальное дописываете вы.

**Вариант Б — перенести готовый:**

```bash
scp .env user@vps:/opt/vpn-service/.env
ssh user@vps 'chmod 600 /opt/vpn-service/.env'
```

### Что должно быть заполнено

Домены — четыре строки, все обязаны совпадать между собой:

```dotenv
APP_DOMAIN=app.vpn-service.help
SUB_DOMAIN=sub.vpn-service.help
PUBLIC_APP_URL=https://app.vpn-service.help
MARZBAN_SUB_URL_PREFIX=https://sub.vpn-service.help
```

Домен написан четырежды не по недосмотру: `PUBLIC_APP_URL` и `MARZBAN_SUB_URL_PREFIX` вмораживаются
в образ на сборке и подстановке во время запуска не поддаются. Расхождение между ними `marzban-init`
ловит до старта и валит деплой — иначе его увидел бы только человек с битым QR.

Остальное:

| Переменная              | Чем заполнить                                                       |
| ----------------------- | ------------------------------------------------------------------- |
| `ACME_EMAIL`            | живая почта — на неё Let's Encrypt шлёт предупреждения об истечении |
| `TELEGRAM_BOT_TOKEN`    | от BotFather, вида `1234567890:AA...`                               |
| `TELEGRAM_BOT_USERNAME` | username бота без `@`                                               |
| `ADMIN_CHAT_ID`         | ваш численный Telegram id; он же единственный админ приложения      |
| `STRIPE_SECRET_KEY`     | `sk_live_…`                                                         |
| `STRIPE_WEBHOOK_SECRET` | `whsec_…`, см. шаг 8                                                |
| `RETURN_DEEPLINK`       | `https://t.me/<бот>/<short_name>` — **не домен**                    |

Найти незаполненное: `grep -n ЗАПОЛНИТЬ .env`

### Чего касаться не нужно

`REALITY_DEST` и `REALITY_SERVER_NAMES` оставьте как есть. Это сайт, под который маскируется
REALITY, и он **специально чужой**: свой домен здесь выдал бы вас — SNI стал бы тем, что цензор
тривиально связывает с вашим IP.

`REALITY_PRIVATE_KEY` и `REALITY_SHORT_ID` оставьте пустыми — `marzban-init` сгенерирует их на
первом старте и сохранит на volume. Заполнять их имеет смысл **только до первого запуска** и только
если хотите держать ключи вне сервера: тогда при пересоздании VPS уже выданные клиентские конфиги
продолжат работать. После первого старта эти переменные игнорируются.

Если решили закрепить свои:

```bash
docker run --rm gozargah/marzban:v0.8.4 xray x25519   # взять Private key
openssl rand -hex 8                                    # shortId, ровно 16 hex-символов
```

В `.env` кладёте только приватный ключ: публичный выводится из него автоматически. Отдельной
переменной для публичного нет намеренно — Marzban не проверяет, что пара сходится, и при
расхождении отдаёт рабочие с виду ссылки, по которым никто не подключится.

## Шаг 5. Запустить

```bash
scripts/deploy.sh
```

Первый прогон занимает 5–15 минут: тянутся образы, ставятся зависимости, компилируется
`better-sqlite3` из исходников.

**Не используйте `docker compose build && up -d`.** Обе половины молча врут:

- `.env` лежит вне контекста сборки, а BuildKit намеренно не кладёт содержимое секрет-маунта в ключ
  кеша. Правка `.env` не меняет ни одного входа сборки — `build` попадает в кеш, и образ остаётся со
  старыми секретами, отрапортовав успех. `deploy.sh` передаёт хеш `.env` build-аргументом.
- `docker compose ps` без `-a` показывает только живые контейнеры, а все три одноразовых сервиса к
  этому моменту уже вышли. Упавший `marzban-check` выглядит там ровно как успешный: никак.
  `deploy.sh` дожидается каждого и падает на его коде возврата.

### Что происходит внутри

| Сервис          | Что делает                                                                                                          |
| --------------- | ------------------------------------------------------------------------------------------------------------------- |
| `app-migrate`   | миграции БД приложения, затем выходит                                                                               |
| `marzban-init`  | сверяет домены → генерирует ключи REALITY и рендерит конфиг Xray → миграции панели → заводит оба админских аккаунта |
| `marzban`       | панель + ядро Xray                                                                                                  |
| `app`           | мини-апп и фоновый воркер                                                                                           |
| `caddy`         | TLS и маршрутизация                                                                                                 |
| `marzban-check` | проверяет, что приложение реально логинится в панель                                                                |

Порядок внутри `marzban-init` задан апстримом, а не вкусом: Marzban разбирает конфиг Xray на этапе
импорта, а миграции и CLI тянут весь пакет приложения — поэтому до появления валидного конфига не
работает ни то, ни другое.

### Успешный финал

```
deploy:   app-migrate: ok
deploy:   marzban-init: ok
deploy:   marzban-check: ok
deploy: done — the app, the panel and the app-to-panel path all check out
```

`marzban-check` — единственное место, где сходится вся цепочка: он логинится в панель тем самым
аккаунтом, что в `.env`, убеждается, что аккаунт не sudo, и что инбаунд с нужным тегом панель
действительно отдаёт. **Его ненулевой выход означает, что оплата пройдёт, а доступ не выдастся.**
Чинить до первой продажи.

## Шаг 6. Проверить сертификаты

```bash
curl -I https://app.vpn-service.help
```

Ждём `HTTP/2 200`. Если висит или ошибка TLS — Caddy ещё не получил сертификат:

```bash
docker compose logs caddy | tail -30
```

## Шаг 7. Зарегистрировать вебхук Telegram

Только **после** того, как сертификат выпущен: Telegram проверяет его прямо в момент регистрации.
Именно поэтому шаг не входит в `deploy.sh` — на первом деплое он проиграл бы гонку.

```bash
docker compose --profile webhook up app-set-webhook
```

Скрипт печатает `getWebhookInfo` с полем `lastError` — там видно просроченный сертификат, `401` от
разъехавшегося секрета или отвалившийся DNS. Ждём `lastError=none`.

Повторять после смены домена или `TELEGRAM_WEBHOOK_SECRET`.

## Шаг 8. Настроить Stripe

В дашборде Stripe заведите вебхук:

- URL: `https://app.vpn-service.help/api/stripe/webhook`
- События: `checkout.session.completed`, `checkout.session.async_payment_succeeded`,
  `checkout.session.async_payment_failed`, `checkout.session.expired`

Полученный `whsec_…` впишите в `STRIPE_WEBHOOK_SECRET` и **пересоберите** — значение вмораживается
в образ:

```bash
scripts/deploy.sh
```

Вебхук Stripe — единственный источник факта оплаты. Редирект на `success_url` фактом не является:
эту ссылку открывают руками.

## Шаг 10. Завести тарифы

На свежей базе тарифов нет — их создаёт админ через интерфейс.

Откройте мини-апп своим аккаунтом (тем, чей id в `ADMIN_CHAT_ID`) → **Профиль** → админка → создайте
тарифы, промокоды и FAQ.

Фикстуры для стенда есть отдельным профилем, но в прод их не заливайте — они добавляют двух
фальшивых пользователей:

```bash
docker compose --profile seed up app-seed
```

## Шаг 11. Включить бэкапы

`tech.md` (A17) требует ночной бэкап с ротацией 14 дней. В `docker compose` он не входит намеренно:
смысл бэкапа — пережить то, что он бэкапит, а задача внутри контейнера умирает вместе с ним.

```bash
apt install -y sqlite3
install -m 700 scripts/backup.sh /usr/local/bin/vpn-backup
crontab -e        # строку взять из scripts/backup.cron.example
```

Складывает в `/var/backups/vpn-service`. **Забирайте копии с сервера наружу** — бэкап на том же
диске не переживает потерю диска.

## Финальная проверка

```bash
docker compose ps -a
```

`caddy`, `app`, `marzban` — `Up`. Три одноразовых — `Exited (0)`.

Дальше руками:

1. Мини-апп открывается из бота, видно приветствие и тарифы.
2. `/start` в боте отвечает ссылкой на приложение.
3. Покупка тестовой картой Stripe (`4242 4242 4242 4242`) доходит до подписки.
4. В профиле появились ссылка подписки и QR.
5. Ссылка импортируется в клиент (Happ, v2rayNG) и **соединение поднимается**.

Пятый пункт — единственный, который проверяет всю цепочку целиком. Пока он не пройден, деплой не
закончен.

## Обслуживание

### Обновить код

```bash
git pull && scripts/deploy.sh
```

### Попасть в дашборд панели

Наружу он не публикуется. Туннель, затем `http://localhost:8000/dashboard/` в местном браузере,
аккаунтом из `MARZBAN_SUDO_USERNAME`:

```bash
ssh -L 8000:127.0.0.1:8000 user@vps
```

### Сменить пароль приложения к панели

Порядок важен: работающее приложение держит **старый** пароль, вмороженный в образ, и прямо сейчас
выдаёт доступы нормально. Сначала пересобрать, потом чинить панель — иначе вы сломаете то, что
работает.

```bash
scripts/deploy.sh                                    # образ с новым паролем
docker compose stop marzban
docker compose run --rm marzban-init marzban-cli admin delete -u vpnapp -y
docker compose up -d marzban-init && docker compose start marzban
```

Панель останавливается не из осторожности: `marzban-init` пишет в тот же файл SQLite, который держит
открытым живая панель.

### Восстановиться из бэкапа

**Сначала volume панели, потом всё остальное**: `marzban-data` держит приватный ключ REALITY и
конфиг Xray. Без них панель поднимется с новыми ключами, и все выданные клиенты умрут.

```bash
docker compose down
B=/var/backups/vpn-service
docker run --rm -v vpn-service_marzban-data:/dst -v "$B":/src:ro alpine \
  sh -c 'rm -rf /dst/* && tar xzf /src/marzban-YYYY-MM-DD.tar.gz -C /dst'
docker run --rm -v vpn-service_app-data:/dst -v "$B":/src:ro alpine \
  cp /src/app-YYYY-MM-DD.db /dst/app.db
scripts/deploy.sh
```

### Логи

```bash
docker compose logs -f app
docker compose logs -f marzban
```

Ротация настроена: 10 МБ × 3 файла на сервис.

## Если сломалось

| Симптом                              | Причина и что делать                                                                                                                                               |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Caddy не получает сертификат         | DNS ещё не смотрит на сервер, либо закрыт `80`. Проверьте `dig`. На время отладки раскомментируйте `ACME_CA` со стейджингом в `.env` — у него мягкие лимиты.       |
| `marzban-init` валится на доменах    | Четыре домена в `.env` разошлись. Сообщение называет, какой именно.                                                                                                |
| `marzban-check`: `401`               | Пароль в `.env` разошёлся с тем, что в панели. См. «Сменить пароль» — **порядок важен**.                                                                           |
| `marzban-check`: инбаунд не найден   | `MARZBAN_INBOUND_TAGS` не совпадает с тегом в конфиге Xray. Побайтово, включая пробелы.                                                                            |
| Оплата проходит, доступа нет         | Смотрите `docker compose logs app` на `subscription.provision`. Почти всегда — панель недоступна или тег инбаунда не тот.                                          |
| Ссылка подписки ведёт на `127.0.0.1` | У контейнера нет исхода в интернет: Marzban определяет внешний IP запросом наружу и при неудаче молча падает на `127.0.0.1`. `marzban-init` предупреждает об этом. |
| Клиент подключается, трафика нет     | `REALITY_DEST` недостижим. Xray стартует нормально и валит каждое рукопожатие — ни один healthcheck этого не видит. `marzban-init` предупреждает.                  |
| Сборка падает по памяти              | Меньше 2 ГБ RAM. Добавьте swap или соберите образ у себя и перенесите через `docker save`.                                                                         |
| Порт REALITY сменили, VPN лёг        | `REALITY_PORT` рендерится в конфиг один раз, а compose публикует его на каждом старте. `marzban-init` теперь валит деплой на расхождении и говорит, что делать.    |

## Чего делать нельзя

- **`docker compose down -v`** — удаляет volume вместе с ключами REALITY и базой заказов. Без `-v`
  они сохраняются.
- **Пушить образ в реестр** — в нём вмороженные секреты.
- **Открывать `8000` наружу** — это дашборд, `/api` и выдача токенов по открытому HTTP.
- **Ставить `REALITY_DEST` на свой домен** — SNI выдаст вас с головой.
- **Менять тег образа Marzban** — `v0.8.4` зафиксирован не для порядка. Ветка `v1.0.0` переписана и
  имеет другие формы API; весь `marzban/` написан против `v0.8.4` построчно. Это перенос интеграции,
  а не обновление.
